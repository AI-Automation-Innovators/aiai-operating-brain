# Start Here for AI Tools

## Purpose
Universal onboarding guide for ChatGPT, Claude, Gemini, Perplexity, NotebookLM, Codex, and future AI tools using this repo.

Use this file to decide what to read first, how to stay factual, and how to avoid changing the repo structure.

## Read first
1. `README.md` — understand what this repo is, who it is for, and what it is not.
2. `START-HERE.md` — follow the general reading path for humans and AI tools.
3. `AGENTS.md` — follow the repository-wide editing rules before making changes.
4. `01-business/` — use as the core source of truth for Kevin Tunis, AI Automation Innovators, voice, offers, and constraints.
5. Relevant folder under `02-projects/` — read project instructions before working on a specific project.
6. `03-public-ai-visibility/` — use when working on AI visibility, entity summaries, visibility prompts, or weekly tracking.
7. `04-proof-and-examples/` — use when writing proof, examples, objections, replies, posts, emails, audit language, or public-facing content.

## How to use this repo
- Treat the existing markdown files as source material.
- Keep answers and edits grounded in the files you read.
- Do not invent facts, proof, pricing, testimonials, client names, guarantees, revenue claims, or private personal details.
- Mark unknown details as `TBD` instead of guessing.
- Preserve the numbered folder structure.
- Keep changes small, practical, factual, and reviewable.

## Tool-specific notes
- ChatGPT: summarize relevant files first, then draft or edit from that context.
- Claude: use the reading order before producing strategy, copy, or project guidance.
- Gemini: cross-check claims against the repo files before expanding or rewriting content.
- Perplexity: use the repo as internal context; use outside sources only when explicitly needed and keep repo facts separate from web findings.
- NotebookLM: load this file, `README.md`, `START-HERE.md`, `AGENTS.md`, and the relevant project or business files as the core source set.
- Codex: read `AGENTS.md` before editing, make only requested markdown changes, and do not add apps, dependencies, workflows, package files, or new folders unless explicitly requested.
