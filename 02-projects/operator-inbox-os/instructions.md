# Operator Inbox OS Instructions

## Purpose
Project-specific instructions for Operator Inbox OS.

Operator Inbox OS is a future-facing project idea for organizing inbound work, messages, tasks, and approvals. It should not distract from the Workflow Leak Audit, which remains the primary revenue lane.

## Current Status
Parked / future-facing.

Do not expand this into a build, app, automation, database, dependency, or workflow system until Kevin explicitly approves scope.

## Goal
Create a practical operating layer for reviewing inbound business information, identifying what needs action, and routing items for human approval.

## MVP Scope
If this project becomes active later, the first version should stay narrow:

- Define inbox sources.
- Define agent roles.
- Define approval rules.
- Define what can be summarized or drafted.
- Define what requires human review.
- Define what is explicitly out of scope.

## Inbox Sources
Potential inbox sources are `TBD` and may include:

- Email.
- Forms.
- Texts.
- CRM notes.
- Calendar items.
- Task lists.
- Internal notes.

Do not assume tool access or integrations are approved.

## Approval Rules
Human approval is required for:

- Sending external messages.
- Making commitments.
- Changing pricing, scope, or terms.
- Handling sensitive customer information.
- Escalating complaints or disputes.
- Triggering automations.
- Updating systems of record.

## What It Will Not Do Yet
- It will not send messages automatically.
- It will not connect to tools.
- It will not create a database.
- It will not become an app.
- It will not replace human judgment.
- It will not make customer promises.
- It will not change business records without approval.

## Why It Should Not Distract From Workflow Leak Audit
Workflow Leak Audit is the clearer near-term offer because it helps identify immediate workflow leakage for local businesses without requiring a build. Operator Inbox OS may become useful later as an implementation direction, but it should remain parked until the audit offer, proof, and delivery process are stronger.

## Update Rules
Keep instructions simple until implementation details are confirmed. Mark unknowns as `TBD`.
